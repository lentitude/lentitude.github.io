Clean Code 第四章：注解

发表于 2019-07-09 | 分类于 Clean Code
字数统计: 672 字 | 阅读时长 ≈ 2 分钟

本文主要讲解 Clean Code 第四章：注解

怎么理解注解：大部分注解是代码的不可阅读性而做的补救措施

如上图所示，注解我理解主要分为三类：

  1. 必要注解
  2. 无用注解
  3. 反作用注解

1. 必要注解

法律信息

公司规范限定的需要在代码信息中加入相对应的法律信息， 比如版本或者著作权的声明。

阐述

方法本身具备一定的复杂性， 对于调用方来讲并不需要过多的了解内部的细节， 或者对该方法的方案选择进行原有的解释。

TODO

方法中存在一些疏漏， 但对于当前方案没有影响。或者对部分细节可以进行再优化， 在代码上进行标注以提醒自己完成相关的功能。

公共API 的 JavaDoc

良好描述的公共API 可以避免使用方关注实现细节， 只需要输入合规的参数， 即可得到对应的结果。 比如标准JAVA 库中的 JavaDoc文档。

无用注释

代码简单， 冗余注释

代码本身的逻辑比较简单， 没必要进行冗余的注释来解释方法或者变量的意义。

循规性注释

每个变量或者方法都进行JavaDoc 的标注或者注释是没有必要的， 除了增加无用的代码行数。

日志性注解

针对每次代码的变动而注释对应的改动内容， 现在可以完全由Git 版本控制工具来完成。

位置标记

为了标记某一个方法位置而进行的注释， 除了增加代码噪音之外， 并不会带来别的好处。使用明显意义的名称更加方便寻找。

括号后面的注释

对于具备长嵌套的代码或许便于开发明确区间，IDE本身就带有这个功能， 最好的方法还是通过缩短代码来解决。

注释掉的代码

对于项目工程中已经注释掉的代码， 不要考虑将来是否会引用，移除掉它。不用担心会丢失这部分数据， 因为都会记录到版本工具中， 已经注释掉的代码会破坏代码的阅读性。

反作用注释

诱导性描述

除了一些本身就存在错误的描述会带来困扰之外， 即便一些开始准确无比的描述， 在后期功能的迭代中也可能会遗漏掉描述的准备性，将会对后续的开发进行误导。
